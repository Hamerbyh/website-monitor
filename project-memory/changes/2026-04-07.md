## Site-Centric Direction
- `sites` should remain the stable parent record for each managed website.
- New capabilities should attach through `site_id` and follow a modular child-table model.
- `issues` should remain the unified abnormal-state entry point regardless of whether the problem comes from uptime, SSL, payments, subscriptions, or another service dependency.
- Prefer structured tables plus limited `jsonb` extension fields over a fully generic schema.

## Candidate Feature Areas
- Site profile and portfolio management:
  - type, business model, priority, owner, tags, environment, notes
- Core health monitoring:
  - HTTP availability, latency, redirect checks, SSL/TLS, DNS, domain expiry
- Service inventory:
  - payments, email, analytics, auth, CDN, storage, search, advertising
- Service health checks:
  - webhook reachability, provider API availability, callback failures, credential expiry
- Critical transaction monitoring:
  - signup, login, checkout, subscription activation, form submission
- Business and growth signals:
  - traffic, search performance, revenue, subscriptions, conversion trends
- Operations workflow:
  - issue triage, alert routing, maintenance windows, change history, per-site status summaries

## Site Type Guidance
- Content sites:
  - uptime, SEO/search signals, analytics trends, sitemap/robots health, ad or affiliate link checks
- Subscription sites:
  - auth flows, checkout, recurring billing state, webhook delivery, subscription status changes
- Commerce or digital product sites:
  - checkout health, payment success/failure trends, callback latency, delivery flow checks
- Tool or app sites:
  - core API health, third-party model/API dependency status, quota or timeout detection
- Lead-generation sites:
  - form submission success, notification email delivery, CRM/webhook connectivity, landing-page performance

## Implementation Guidance
- Keep repo-level guidance and examples appropriate for an open-source codebase, even when product behavior is currently personal-use oriented.
- Avoid adding provider-specific booleans such as `stripe_enabled` directly to `sites`.
- Add `site_type`, `business_model`, and similar stable classification fields early if they are needed for filtering and dashboard grouping.
- Favor new child entities such as `site_services`, `site_service_status`, `site_service_checks`, and transaction-monitoring tables for the next expansion phase.
- Keep homepage and future dashboards focused on "which sites need action now" rather than charts first.
